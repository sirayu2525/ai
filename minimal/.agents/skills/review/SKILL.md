# Review

## 使うとき

- 既存差分、PR、ファイル、文書のレビューをしたいとき。
- 正しさ、回帰リスク、検証漏れ、改善余地を見つけたいとき。

## 進め方

1. レビュー対象と重視する観点を確認する。
2. 正しさの問題、回帰、セキュリティ、データ破壊、テスト不足を優先して見る。
3. 必要に応じて設計、責務分離、命名、可読性まで広げる。
4. findings を重要度順に返し、指摘がない場合も残るリスクを伝える。

## 注意

- 指摘はファイルや該当箇所に結びつける。
- 推測で断定せず、前提がある場合は明示する。
