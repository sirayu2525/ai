# Implementation Workflow

## 着手前

- 依頼内容、関連ファイル、リポジトリ状態から文脈を組み立てる。
- コードベースだけでは判断できない点は前提として明示する。
- 多段階またはリスクのある作業では短い計画を先に置く。
- 目先の修正だけでなく、保守性、拡張性、読みやすさを含めてより良い実装方針を最初に検討する。

## 実装中

- 既存実装に引きずられず、必要なら抽象化、責務の分離、データや処理の流れの再設計を行う。
- 部分修正よりも全体設計を見直した方が良いなら、ゼロベースで実装方針を組み替えてよい。
- 変更範囲は目的なく広げないが、コードを綺麗にするために必要な再構成はためらわない。
- その場しのぎの修正ではなく、問題の原因に対して筋の良い解決を優先する。
- リファクタや再設計は、コード品質を上げるために意味があるなら作業に含めてよい。
- 挙動が変わるなら周辺ドキュメントやコメントも更新する。

## 検証

- まずは狭く有効な確認から行い、必要なら広い検証に進む。
- 単に動くかだけでなく、設計、可読性、境界条件、保守性まで含めて妥当かを確認する。
- 検証できなかった場合はその事実を明示する。
- 断定する代わりに、残るリスクや次の確認点を示す。

## 締め方

- 変更内容をユーザー影響ベースで短くまとめる。
- 実施した検証と、未実施の確認を分けて伝える。
- 前提に依存する場合は、残るリスクを添える。
