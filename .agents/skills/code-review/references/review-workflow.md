# Review Workflow

## 着手前

- 依頼内容、差分、関連ファイルからレビュー対象を特定する。
- 実装提案に進む前に、まず現状の変更で何が危ないかを見つける。
- 前提が足りない場合は、推測と事実を分けて扱う。

## まず見る点

- 正しさのバグ、回帰、検証漏れ、テスト不足を優先して見る。
- API 互換性、設定変更、権限、秘密情報、破壊的変更の有無を確認する。
- 必要に応じて責務分離、命名、将来の変更コスト、不自然な設計も確認する。

## まとめ方

- findings を最優先で並べる。
- 可能なら具体的なファイルと行番号を示す。
- 指摘がない場合も、そのことを明示する。
- 断定に前提がある場合は、残るリスクや追加確認点を添える。
