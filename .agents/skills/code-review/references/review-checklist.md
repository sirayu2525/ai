# レビューチェックリスト

## まず見る点

- 正しさのバグ、回帰、検証漏れ、テスト不足を優先して見る。
- 忖度せず、読みやすさ、保守性、責務分離、命名、設計の妥当性も品質基準として確認する。
- 可能なら具体的なファイルと行番号を示す。
- 指摘事項と、要約や未確定事項を分けて書く。

## リスク確認

- 境界値、空入力、失敗系の処理を確認する。
- schema、config、API 互換性の変更を確認する。
- 権限、秘密情報の扱い、破壊的操作を確認する。
- 変更後の挙動をテストがまだ十分カバーしているか確認する。
- 動作上は成立していても、将来の変更コストが高い実装や不自然な設計がないか確認する。

## まとめ方

- 指摘がない場合も、そのことを明示する。
- 妥協ではなく良いコードを目指す観点で判断し、改善余地があれば遠慮なく指摘する。
- 前提に依存する場合は、残るリスクやカバレッジの穴を添える。
