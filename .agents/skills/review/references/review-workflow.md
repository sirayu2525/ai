# Review Workflow

## 着手前

- レビュー対象とレビュー観点を確認する。
- 指摘が求められているのか、改善提案も求められているのかを分ける。
- 対象がコードなら、正しさと回帰リスクを最優先に見る。

## まず見る点

- 正しさのバグ、回帰、検証漏れ、テスト不足を優先する。
- 必要に応じて設計、責務分離、命名、可読性、将来の変更コストまで広げる。
- 断定に前提がある場合は、その前提を明示する。

## `research` への横移動

- より良い実装方法、適切なライブラリ、現在のベストプラクティス、設計上の代替案が重要な場合は、`research` に横移動してよい。
- ただし主目的はレビューなので、調査はレビュー品質を上げるための補助に留める。
- `research` から戻るときは、確認した事実、比較した選択肢、推奨案、未確定事項を持ち帰る。
- findings と、参考として調べた改善案やベストプラクティスは分けて返す。

## `verification` への横移動

- 既存の説明や diff だけでは足りず、実際のテスト結果、ビルド結果、挙動確認が必要なら `verification` に横移動してよい。
- `verification` の主責務は確認を実行して証拠を返すことであり、レビューの結論そのものは `review` で出す。
- `verification` から戻るときは、実施した確認、結果、未実施の確認を持ち帰る。

## まとめ方

- findings を最優先で並べる。
- 可能ならファイルと行番号や対象箇所を示す。
- 指摘がない場合も、そのことと残るリスクを明示する。
